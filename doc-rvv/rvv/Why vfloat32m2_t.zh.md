# RVV 向量类型选型

在 x86 架构的 AVX/SSE 指令集中，向量寄存器宽度是固定的（例如 `__m256` 代表 256 位，打包 8 个单精度浮点数）。RISC-V Vector (RVV) 扩展引入了 VLEN（向量寄存器长度）不可知的设计，并通过 LMUL（向量寄存器组乘数）机制，允许开发者动态调整数据组织方式。

在实现 PCL 的 RVV 优化时，需要在吞吐量、寄存器压力以及数值精度之间进行权衡。经过性能测试对比，最终确立了以 `vfloat32m2_t` 为主的向量化策略。

## 1. 寄存器压力与流水线吞吐的权衡：确立 LMUL=2

在确定主数据类型时，需要为单精度浮点数（32-bit）选择合适的 LMUL 值。RVV 允许将多个向量寄存器捆绑（如 m2, m4, m8）以提高单指令数据吞吐量，但这会成倍消耗物理向量寄存器（总数固定为 32 个）。

以数学函数 `acos` 为例，其内部包含多项式近似计算。执行期间，流水线需要同时维护输入变量 `x`、中间结果 `mul_term`，以及至少 8 个常数系数向量（`a0` 至 `a2`, `b0` 等）。

- **采用 m8（捆绑 8 个寄存器）：** 32 个物理寄存器最多只能同时驻留 4 个逻辑变量（32 / 8 = 4）。这会导致编译器在计算 `acos` 时触发寄存器溢出（Spilling），频繁将中间数据换入换出堆栈，访存开销会直接抵消 SIMD 带来的性能收益。
- **采用 m1（单寄存器）：** 虽规避了溢出，但单指令处理的数据量小，无法有效掩盖 RVV 硬件在指令发射（Issue）和解码阶段的固定延迟，导致流水线利用率偏低。

**最终决策：** 选用 m2（即 `vfloat32m2_t`）作为折中方案。该配置将单个变量的寄存器占用控制在 2 个，系统可同时维护约 16 个激活的向量变量。这足以覆盖 `acos` 及其调用链路所需的常数和中间状态，在实现零 Spilling（全寄存器计算，无内存读写开销）的同时，提升了单指令吞吐量。

## 2. 数值精度策略：Float 与 Double 的流向控制

PCL 多数点云类型（如 `pcl::PointXYZ`）的坐标数据类型为 float，但在部分距离计算、模型拟合的底层标量实现中，会在累加或结果存储阶段转为 double（如写入 `std::vector<double>` 容器）。

在 RVV 向量化设计中，存在两种数据流向：

- **全程 `vfloat64m2_t`：** 在 Load 阶段将点坐标宽化为 double，后续所有数学核（如欧氏距离、夹角计算）全程走双精度向量运算。
- **计算 `vfloat32m2_t`，写回宽化：** 维持 float 精度完成计算，仅在循环末尾需要兼容 PCL double 容器时，执行一次性宽化写回。

对于 `selectWithinDistance`、`getDistancesToModel` 等基于 RANSAC 的鲁棒估计场景，算法本身对单次距离计算的双精度依赖不高。因此，当前实现选择了后者。

该方案的数值行为与既有的 x86 SSE/AVX 向量化实现保持一致（误差控制在 $10^{-3} \sim 10^{-2}$ 量级，主要源于 `acos` 近似与单精度截断）。同时，`vfloat32m2_t` 单次处理的元素数量是 `vfloat64m2_t` 的两倍，且省去了高频的浮点类型转换指令开销。此外，复用单精度数学库（_f32m2）也降低了后续的维护成本。

## 3. 规避 SLF 性能停顿：即转即存（Convert-and-Store）机制

RVV 1.0 规范要求：执行宽化指令时，目标向量寄存器的 LMUL 必须与元素宽度（SEW）同步翻倍。即源寄存器为 `vfloat32m2_t` 时，目标寄存器必须是 `vfloat64m4_t`。

为规避 m4 带来的寄存器压力，早期测试方案尝试了内存中转：先将 float 向量 Store 到栈临时数组，再用标量 `static_cast<double>` 读出并写入目标地址。但这种读写位宽不一致的操作可能会触发 Store-to-Load Forwarding (SLF) 冲突，导致 CPU 乱序执行流水线发生较长周期的停顿。

因此，最终代码摒弃了内存中转，采用“即转即存”的寄存器内宽化策略：在循环体的最后一步，直接调用 `vfwcvt` 将结果宽化至 m4 寄存器组，紧接着调用 `vse64` 写入主存。由于 m4 变量的生命周期仅限于循环末尾，存入内存后即被释放，因此不会对循环体前中段的数学计算产生寄存器分配干扰。

## 4. 掩码类型的硬件约束：vbool16_t 推导

确定主数据类型为 `vfloat32m2_t` 后，掩码类型的选择由 RVV 硬件规范直接决定。RVV 掩码位宽（MLEN，即单 bit 对应的数据位宽）严格遵循以下公式：

$$\text{MLEN} = \frac{\text{SEW}}{\text{LMUL}}$$

代入本工程配置：元素位宽（SEW）为 32-bit，寄存器组乘数（LMUL）为 2，得出 MLEN = 16。这意味着掩码向量中的每 1 bit 对应逻辑长度为 16-bit 的数据块。因此，编译器内建函数（如比较指令 `vmflt_vf_f32m2_b16`）的掩码后缀为 b16，其对应的 C 语言类型必须是 `vbool16_t`。这一硬件设计确保了底层 VLEN 变化时，掩码与数据的映射保持对齐。
